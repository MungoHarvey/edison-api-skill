## Problem Statement

The Edison Scientific API wrapper has polished Python scripts and SKILL.md definitions
for six job types (literature search, precedent checks, molecular design, dataset
analysis, async batch, evaluation). But it's not packaged as a Claude Code plugin —
meaning there's no plugin.json, no skills/ directory structure, and no install path
for other people using Claude Code for science.

The goal: turn this repo into a proper Claude Code plugin so anyone can install it
with `cc --plugin-dir /path/to/repo` and get Edison's capabilities surfaced as
natural-language skills inside Claude Code, with zero script-path knowledge required.

## What Makes This Cool

Zero friction: you describe your scientific question in natural language, Claude picks
the right job type (LITERATURE, PRECEDENT, MOLECULES, ANALYSIS), invokes the
adjacent Python script, and returns results inline. No CLI knowledge needed.

Sharable: anyone building AI-assisted scientific research workflows in Claude Code
can install this plugin and get immediate Edison capability.

## Constraints

- Platform: Windows (Git Bash / bash shell). Symlinks unreliable — avoid.
- Existing scripts stay in place (edison-*/scripts/). No moves there.
- `uv` required for Python environment management (not pip/conda/venv).
- CLAUDE.md paths and documentation should remain valid after restructure.

## Premises

1. The existing SKILL.md files in edison-literature/, edison-molecules/, etc. are
   the skill definitions — they just need restructuring into skills/<name>/SKILL.md
   format, wrapped with plugin.json.
2. The plugin lives in this same repo — `.claude-plugin/plugin.json` at repo root
   makes it installable as a Claude Code plugin without a separate package.
3. "Make it available" means a GitHub repo others can install by cloning — not
   a marketplace submission (that can come later).
4. Python scripts stay in their current locations. Plugin skills invoke them by
   relative path from the project root.
5. No UI component. Pure structure and manifest work.

## Approaches Considered

### Approach A: Restructure into plugin-native layout (CHOSEN)
Move/rename SKILL.md files to fit `skills/<name>/SKILL.md` format. Add plugin.json
at root. Existing scripts stay in place. Effort: S. Risk: Low.

### Approach B: Add plugin.json + symlinks (REJECTED)
Keep everything as-is, add thin wrapper skills/. Rejected: symlinks break on Windows.

### Approach C: Flat skills/ layout, scripts stay (NOT CHOSEN)
Parallel SKILL.md locations — ongoing maintenance burden with two docs to keep in sync.

`.claude-plugin/plugin.json` shape (verified against official plugin-dev schema):
```json
{
  "name": "edison-scientific",
  "description": "Edison Scientific API skills for Claude Code — literature search, molecular design, precedent checks, and biological dataset analysis.",
  "author": {
    "name": "Mungo Harvey",
    "email": "mungo@the-harveys.org"
  }
}
```

Note: no `version` field — verified against real plugin.json examples. Skills are
auto-discovered from `skills/<name>/SKILL.md` — no explicit skills array in manifest.

**Script path references in SKILL.md:** Claude Code invokes Bash commands from the
user's project root (not from the skill directory). Script paths in SKILL.md like
`.venv/bin/python edison-literature/scripts/literature_search.py` reference paths
relative to the project root and remain valid after the SKILL.md moves to
`skills/edison-literature/SKILL.md`.

**Windows path note:** Scripts use `.venv/bin/python` (Unix-style path). This works
in Git Bash. If Claude Code invokes them through a different subprocess, the path
may need to be `.venv/Scripts/python.exe` on Windows. Verify on first install.

## Open Questions

1. Should `edison-setup` be a skill (user-invokable as `/edison-scientific:edison-setup`)
   or documentation only? RECOMMENDATION: Include it — users hitting import errors
   will want a self-healing setup command. Label description: "Run setup and validate
   Edison environment."
2. Should SKILL.md files get updated `description` frontmatter with natural-language
   trigger phrases, or keep existing content verbatim? RECOMMENDATION: Update them —
   adding trigger phrases like "When user asks a literature search question, invoke
   this skill" improves Claude's automatic routing.

## Success Criteria

- `cc --plugin-dir /path/to/edison-api-skill` loads all skills without error
- `/edison-scientific:edison-literature` appears in `/help`
- Running a skill invokes the correct Python script with correct args
- Someone new could install and use it without reading the scripts
- plugin-validator agent passes with no critical errors
