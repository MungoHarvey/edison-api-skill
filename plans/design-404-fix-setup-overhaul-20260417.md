## Problem Statement

The Edison Scientific Claude Code plugin shipped its structural restructure (skills/
layout, plugin.json). Three issues now block it from being reliably usable:

1. **404 on result retrieval** — `run_tasks_until_done` fails mid-poll. All PEP 723
   dep blocks pin `"edison-client"` with no version, so uv picks whatever's latest
   or cached — which may call a deprecated endpoint.

2. **7 setup friction points** from a post-install retrospective on Windows:
   - `setup_venv.sh` hardcodes `.venv/bin/python` (breaks on Windows)
   - `uv venv` picks Python 3.10 by default; `edison-client>=0.9.0` requires 3.11
   - Install path underdocumented — clone alone doesn't register skills in Claude Code
   - `.env` walk-up stops at any README.md/VENV, not at actual project root (`.git`)
   - No project-level install story for repos that already have `.venv` and `.env`
   - `--ping` endpoint returns 404 (server-side, but needs graceful handling)
   - Error messages point at symptoms not causes (no OS detection, no version hints)

3. **Edison docs MCP not connected** — the Edison Scientific API documentation is
   available as an MCP source but not registered, so Claude can't self-reference the
   API when working on this codebase.

## What Makes This Cool

A user installs this plugin and immediately gets reliable scientific AI in their
Claude Code workflow — literature search, molecular design, precedent checks — with
no knowledge of the underlying API, no path debugging, no version detective work.
The setup should be three steps: install uv, add API key, uv run.

## Constraints

- Windows (Git Bash) is the primary dev environment
- All existing scripts and their relative paths stay as-is
- `uv` is required and available; `uv run` with PEP 723 is the primary path
- Plugin structure (`skills/<name>/SKILL.md`) already correct — no structural changes

## Premises

1. The 404 is at least partly client-side: unpinned `edison-client` may resolve to
   a version calling a deprecated endpoint. Fix by pinning a minimum version in all
   PEP 723 dep blocks. Use the Edison docs MCP to confirm the right version.
2. `setup_venv.sh` is not needed for `uv run` — scripts auto-install their deps.
   Demote it to secondary path; "install uv, set key, uv run" is the primary story.
3. The `.env` walk-up should stop at `.git` (first parent containing `.git`), not at
   any directory containing README.md or .venv. `.env.edison` checked first.
   `EDISON_ENV_FILE` env var supported for explicit override.

### 3. Fix .env walk-up: Stop at .git, add .env.edison + EDISON_ENV_FILE

In `skills/edison-setup/scripts/check_environment.py`, update the root detection
(the `find_project_root` function) to:

a) Check `EDISON_ENV_FILE` env var first — if set, use that file directly.
b) Walk up from the script looking for `.env.edison` first, then `.env`.
c) Stop walking when `.git` is found — that's the project root.
d) Do NOT stop at a directory just because it contains README.md or .venv.

Same logic should be applied to the `.env` walk-up in all scripts (the `for _ in
range(8)` loop in each script's preamble).

Updated preamble pattern for all 7 scripts listed in §1:
```python
try:
    from dotenv import load_dotenv
    _env_file = os.environ.get("EDISON_ENV_FILE")
    if _env_file:
        load_dotenv(_env_file)
    else:
        root = Path(__file__).resolve().parent
        for _ in range(8):
            if (root / ".env.edison").exists():
                load_dotenv(root / ".env.edison")
                break
            if (root / ".env").exists():
                load_dotenv(root / ".env")
                break
            if (root / ".git").is_dir():
                break  # at project root, no .env found
            root = root.parent
except ImportError:
    pass
```
Note: `setup_venv.sh` does not use this pattern and is not affected by this change.

`test_connection.py` also needs this update (it has the same old walk-up on lines 20-26).
Additionally, `test_connection.py` lines 50-54 need the same 404-graceful handling from §5:
replace the bare `sys.exit(1)` on platform failure with a 404-aware check that prints a
warning instead of failing setup.

### 4. Add install.sh + install.ps1

Two thin installer scripts at repo root.

Claude Code discovers skills from `~/.claude/skills/<skill-name>/SKILL.md` (user-level)
or `.claude/skills/<skill-name>/SKILL.md` (project-level). The `--user` path copies to
the user-level location. Source: Claude Code plugin documentation (skills/ flat layout).

**install.sh** (Unix/Mac/Git Bash):
```bash
#!/usr/bin/env bash
# Edison Skills installer
# Usage: bash install.sh --user       # copies to ~/.claude/skills/
#        bash install.sh --plugin-dir  # prints the --plugin-dir flag to use
set -euo pipefail
REPO_ROOT="$(cd "$(dirname "$0")" && pwd)"
case "${1:-}" in
  --user)
    mkdir -p ~/.claude/skills
    for skill in "$REPO_ROOT"/skills/edison-*; do
        name=$(basename "$skill")
        if [ -d "$HOME/.claude/skills/$name" ]; then
            echo "WARNING: $name already exists at ~/.claude/skills/$name — overwriting."
        fi
        cp -r "$skill" ~/.claude/skills/ || { echo "ERROR: failed to copy $name — check permissions on ~/.claude/skills/" >&2; exit 1; }
        echo "Installed $name"
    done
    echo "Done. Skills available in Claude Code."
    ;;
  --plugin-dir|*)
    echo "Add this flag to your Claude Code command:"
    echo "  --plugin-dir '$REPO_ROOT'"
    ;;
esac
```

**install.ps1** (Windows PowerShell):
```powershell
# Edison Skills installer for Windows
# Usage: .\install.ps1 -User    # copies to %USERPROFILE%\.claude\skills\
#        .\install.ps1 -PluginDir  # prints the --plugin-dir flag to use
param([switch]$User, [switch]$PluginDir)
$RepoRoot = Split-Path -Parent $MyInvocation.MyCommand.Path
if ($User) {
    $dest = "$env:USERPROFILE\.claude\skills"
    New-Item -ItemType Directory -Force -Path $dest | Out-Null
    Get-ChildItem "$RepoRoot\skills\edison-*" | ForEach-Object {
        $name = $_.Name
        if (Test-Path "$dest\$name") {
            Write-Warning "$name already exists at $dest\$name — overwriting. Customisations will be lost."
        }
        Copy-Item $_.FullName $dest -Recurse -Force
        Write-Host "Installed $name"
    }
    Write-Host "Done. Skills available in Claude Code."
} else {
    Write-Host "Add this flag to your Claude Code command:"
    Write-Host "  --plugin-dir '$RepoRoot'"
}
```

### 5. Graceful --ping failure

The `v0.1/crows` endpoint returns 404 server-side. Until fixed upstream, Check 5
should not exit 1 on 404 — it should distinguish "404 from known endpoint" (warn,
don't fail) from "network error / auth failure" (fail).

Full replacement for `ping_platform` in `check_environment.py`:

```python
def ping_platform(client_class):
    """Check 5 (optional --ping): Submit DUMMY task to verify connectivity.
    Returns True (success), False (hard failure), or None (indeterminate/404).
    """
    api_key = os.getenv("EDISON_API_KEY")
    if not api_key:
        return False
    try:
        client = client_class(api_key=api_key)
        response = client.run_tasks_until_done({"name": "DUMMY", "query": "ping"})
        return True
    except Exception as e:
        msg = str(e).lower()
        if "404" in msg or "not found" in msg:
            print("  ⚠ Ping endpoint returned 404 (known platform issue).", file=sys.stderr)
            print("    Your API key and setup are likely fine.", file=sys.stderr)
            print("    This is a server-side issue at platform.edisonscientific.com.", file=sys.stderr)
            return None  # indeterminate — do not fail setup
        if "401" in msg or "unauthorized" in msg or "forbidden" in msg:
            print(f"  ✗ Authentication failed: {e}", file=sys.stderr)
            print("    Check that EDISON_API_KEY is correct.", file=sys.stderr)
        else:
            print(f"  Connectivity check failed: {e}", file=sys.stderr)
        return False
```

Caller in `main()` — update the ping result handling:
```python
ping_ok = ping_platform(client_class)
if ping_ok is True:
    print("  ✓ Platform connection confirmed", file=sys.stderr)
elif ping_ok is None:
    print("  ⚠ Platform connectivity indeterminate (404 on ping endpoint)", file=sys.stderr)
    # Do NOT sys.exit(1) — this is a known server-side issue
else:
    print("✗ Platform connectivity failed", file=sys.stderr)
    sys.exit(1)
```
